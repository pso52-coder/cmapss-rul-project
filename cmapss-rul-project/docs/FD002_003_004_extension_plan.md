# FD002~FD004 확장 계획 — C-MAPSS

> FD001을 레퍼런스로 완주한 뒤, FD002~FD004로 확장할 때의 차이점과 주의사항. **지금 FD002~004 분석을 시작하지 않으며**, 확장 시 참고할 계획 문서다.

## 0. 확장 전제

FD001 현재 결과는 파이프라인 레퍼런스이며, 최적 모델 탐색 완료본은 아니다.
따라서 FD002~FD004 확장 전에 FD001에서 다음 실험을 우선 수행한다.

- XGBoost / LightGBM 비교
- RUL clipping cap 비교
- 고장 임박 구간 sample weighting
- validation 기반 threshold optimization
- GroupKFold 기반 엔진 단위 검증

이 과정을 통해 FD001 기준 최종 전략을 확정한 뒤,
동일한 실험 프레임워크를 FD002~FD004에 적용한다.

## 1. FD001과 FD002~FD004의 차이 (왜 그대로 적용하면 안 되는가)

| 구분 | FD001 | FD002 | FD003 | FD004 |
|---|---|---|---|---|
| 운전 조건 | 1개 | 6개 | 1개 | 6개 |
| 고장 모드 | 1개(HPC) | 1개 | 2개(HPC+Fan) | 2개 |

FD001은 가장 단순한 단일 조건·단일 고장모드다. 다른 셋은 운전 조건이 6개로 늘거나 고장 모드가 2개로 늘어, **FD001에서 내린 전처리·모델·운영 결정이 그대로 유효하지 않다.**

## 2. 그대로 적용하면 안 되는 항목

### 2.1 op_setting 제외 금지
- FD001은 단일 조건이라 op_setting을 제외했다.
- **FD002·FD004는 운전 조건이 6개**라 op_setting이 핵심 정보다. 제거하지 말고, 모델 입력으로 쓰거나 **운전 조건별(클러스터별) 정규화** 기준으로 활용한다.

### 2.2 무변화 센서 목록 재결정
- FD001의 무변화 센서(sensor_1,5,6,10,16,18,19)를 그대로 쓰지 말 것.
- 운전 조건이 여러 개면 조건에 따라 센서 거동이 달라져, **상수처럼 보이던 센서가 정보가 될 수 있다.**
- 각 FD에서 **EDA를 다시 수행**해 무변화 센서를 재판단한다.

### 2.3 정규화 방식 재검토
- 다중 운전 조건에서는 단순 전체 MinMax가 부적절할 수 있다.
- **운전 조건별 정규화(condition-based normalization)** 를 검토한다. (조건을 먼저 군집화한 뒤 조건 내에서 스케일링)

### 2.4 RUL 클리핑 cap 재결정
- FD001 cap=125는 FD001 수명 분포(최소 128) 기준이다.
- 각 FD의 엔진 수명 분포를 다시 보고 cap을 정한다.

### 2.5 threshold=40 적용 금지
- threshold=40은 FD001 test 기준 후보 운영값이다.
- 다중 고장모드/조건에서는 예측 분포가 달라지므로, **각 FD에서 threshold별 분석을 다시** 한다.

## 3. 확장 시 반복할 절차

각 FD(002/003/004)에 대해 [FD001_reference_workflow.md](FD001_reference_workflow.md)의 Step 01~06 흐름을 반복한다. 단, 위 2장의 항목은 FD별로 새로 판단한다.

1. 데이터 로딩 (sep=r'\s+', 26컬럼) — 동일
2. RUL 생성 (max_cycle − cycle) — 동일
3. **EDA 재수행** — 무변화 센서·상관·운전조건 분포 새로 확인
4. **전처리 재결정** — op_setting 포함 여부, 조건별 정규화, cap
5. 모델링 (baseline부터) — 구조 동일, 데이터만 교체
6. 평가 (test 마지막 cycle, FNR 중심) — 동일
7. 개선 (약점 진단 후 시계열/조건 피처) — FD별 약점에 맞게

## 4. 추가로 고려할 점

- **다중 고장모드(FD003/004)**: 단일 회귀로 두 고장모드를 함께 학습하면 성능이 떨어질 수 있다. 고장모드 구분 또는 모드별 모델을 검토.
- **운전 조건 군집화(FD002/004)**: op_setting으로 6개 조건을 군집화한 뒤 조건을 피처로 추가하거나 조건별 정규화.
- **교차 일반화 검증**: FD001에서 만든 모델을 다른 FD에 바로 적용하지 말고, 각 FD에서 재학습·재평가.

## 5. 결론

FD001은 "가장 쉬운 기준점"이다. 여기서 만든 코드 구조와 단계 흐름은 재사용하되, **데이터 특성에 따른 결정(센서 제거·정규화·op_setting·cap·threshold)은 FD마다 EDA로 다시 내린다.** 이 원칙을 지켜야 확장이 신뢰할 수 있다.
